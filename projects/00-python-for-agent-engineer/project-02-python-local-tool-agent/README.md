# Project 02：Python Local Tool Agent

## 项目目标

做一个本地工具驱动的最小 Agent Runner，把 Python 基础正式映射到 Agent 代码形态。

## 什么时候开始做

只有在完成下面内容后再开始：

- `Project 01`
- `lectures/00-python-for-agent-engineer/07`
- `labs/00-python-for-agent-engineer/07`
- `notes/00-python-for-agent-engineer/study-notes.md` 已补到 lecture 07 和 lab 07

## 用户场景

用户输入一个任务，程序：

1. 接收输入
2. 判断调用哪个本地工具
3. 记录每一步
4. 输出最终结果
5. 在失败时保留足够调试信息

## 架构说明

建议至少拆成这些模块：

- `main.py`：运行入口
- `agent.py`：调度主流程
- `tools.py`：本地工具集合
- `state.py`：状态对象或状态结构
- `validators.py`：输入与结果验证
- `logger.py`：运行记录

## 必做能力点

- 至少 2 个本地工具
- 工具输入校验
- 最小状态记录
- 最大轮次或明确停止条件
- 工具调用日志
- 最终结果和中间步骤分离

## 运行方法

你需要在项目目录里写清楚：

- 如何运行
- 输入长什么样
- 工具会如何被调用
- 输出里哪些是最终结果，哪些是调试信息

## 评估方法

至少验证下面五种情况：

1. 正常命中工具 A
2. 正常命中工具 B
3. 输入为空或参数非法
4. 工具内部报错
5. 多轮或重复调用时是否能正确停止

## 已知问题

这一步要主动写：

- 当前没有模型决策时，你是怎么模拟调度的
- 现在的状态结构哪里还很脆弱
- 哪些地方一旦扩展就需要更严格 schema

## 后续迭代方向

- 引入真实模型决策层
- 引入结构化 schema
- 加测试和 run traces
- 为后续 eval 做样本记录

## 项目完成标志

- 你能解释工具边界为什么这样定
- 你能说清楚状态结构为什么这样设计
- 你知道一个最小 Agent 为什么首先会坏在控制流、验证和日志
