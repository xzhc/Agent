# 外部材料分析策略

这份文档定义当学习过程中需要阅读、分析或复现外部材料时，默认应该怎么选工具。目标不是一开始就接很多外部能力，而是用最小成本拿到足够信息。

## 默认决策顺序

遇到外部材料时，默认按这个顺序判断：

1. 先用网络搜索和网页阅读。
2. 如果是代码仓库，再考虑拉到本地分析。
3. 只有当访问外部系统成为稳定高频需求时，再考虑 MCP。

这个顺序适用于大多数学习任务，也符合“干中学、按需引入”的原则。

## 什么时候直接用网络搜索和网页阅读

下面这些场景，通常直接用搜索、打开网页和阅读页面内容就够了：

- 阅读博客、官方文档、产品说明、README
- 初步判断一个 GitHub 项目值不值得深入
- 阅读少量 issue、PR、release note 或讨论
- 一次性分析某篇文章、某个公开页面、某个仓库首页

适合解决的问题：

- 这是什么
- 值不值得继续投入时间
- 它大概解决什么问题
- 我应该先看哪一部分

## 什么时候优先拉到本地分析

如果目标已经从“了解一下”变成“真正学习或复现”，通常本地分析比纯网页阅读更高效。

下面这些场景优先考虑拉到本地：

- 需要全局搜索代码结构
- 需要跨多个文件理解实现逻辑
- 需要运行 demo、安装依赖、补实验
- 需要改代码、加注释、做最小复现
- 需要对一个开源项目做系统性拆解

对 GitHub 仓库来说，常见顺序是：

1. 先看仓库首页和 README
2. 确认值得深入后再 clone 到本地
3. 用本地搜索、运行和修改来完成真正分析

## 什么时候才考虑 MCP

MCP 适合的是“持续访问某个外部系统”的场景，不适合拿来替代普通网页阅读。

只有出现下面情况时，才值得考虑配置相应 MCP：

- 你需要稳定访问某个平台的数据或操作能力
- 没有 MCP 时，需要反复手工复制粘贴信息
- 这个外部连接会在很多任务里复用，而不是一次性需求
- 需要结构化、可信、低摩擦地取数据或执行操作
- 可能涉及账号态、私有数据或高频查询

如果只是偶尔读公开内容，通常还不到上 MCP 的时机。

## GitHub 的默认策略

对 GitHub，默认策略是：

1. 先看仓库主页、README、文档、issue、PR 等公开页面
2. 真正要学习代码时，把仓库拉到本地
3. 只有当你频繁需要仓库级搜索、PR/issue 流程访问、发布信息跟踪或私有仓库能力时，再考虑 GitHub 相关 MCP

简单说：

- 看资料：网页就够
- 读代码：本地 clone 更好
- 高频连接 GitHub 工作流：再考虑 MCP

## X 的默认策略

对 X 这类平台，默认更保守：

- 如果只是偶尔读公开帖子或线程，先用搜索和网页阅读
- 如果网页可访问性差、结构不稳定、信息噪声大，不要强行把它变成主要学习入口
- 只有当你真的持续依赖它做信息收集或监控时，才考虑专门连接方式

对学习仓库来说，X 更适合作为补充线索，不适合作为主资料来源。

## 一个简单判断清单

碰到外部材料时，可以先问自己这五个问题：

1. 我现在是想“了解”，还是想“复现”？
2. 这个材料是网页内容，还是代码仓库？
3. 这次需求是一次性的，还是会重复很多次？
4. 没有额外连接能力时，我的主要成本是阅读成本，还是搬运成本？
5. 这个外部来源是否值得进入长期工作流？

对应决策通常是：

- 以阅读为主：先用搜索和网页
- 以代码理解和复现为主：拉到本地
- 以高频外部连接为主：再考虑 MCP

## 仓库默认原则

在这个仓库里，遇到外部材料时默认遵循下面的顺序：

- 不先为“可能以后有用”配置 MCP
- 不把网页阅读问题过度工程化
- 不用 MCP 替代本地代码分析
- 先用最小方案完成当前任务，再看是否值得升级工作流
