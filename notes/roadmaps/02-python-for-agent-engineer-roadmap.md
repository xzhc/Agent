# Python 基础学习路线图（面向 Agent Engineer）

## 目的

这份路线图不是为了把 Python 学成一门“面面俱到的通用语言”，而是为了尽快建立后续 Agent 开发真正需要的 Python 能力：

- 能读懂和修改最小 LLM / Agent 项目
- 能自己写脚本、封装函数、组织模块
- 能处理 API、JSON、文件、异常和配置
- 能支撑后续的 tool calling、agent loop、eval 和调试

这份路线图默认服务于“成为 Agent Engineer”，所以强调取舍，不追求语法全覆盖，也不追求把 Python 学成算法竞赛或后端开发全家桶。

## 设计原则

- 先学能支撑后续实验和项目的高频部分。
- 先会读、会改、会跑，再追求写得优雅。
- 先掌握脚本和模块化，再碰复杂框架。
- 每一阶段都要落到真实产出，不只做语法题。
- 学 Python 的同时，逐步学会用 Codex 协助读代码、改代码、排错。

## 学习边界

### 现在优先学什么

- 变量、条件、循环、函数、模块这些最基本控制结构
- `list / dict / set / tuple` 这些常用数据结构
- JSON、文件读写、异常处理、环境变量
- HTTP API 调用、请求响应、状态码和超时
- 适合 LLM / Agent 场景的代码组织方式

### 现在不用展开学什么

- 复杂面向对象设计
- 元类、描述符、装饰器高级玩法
- 算法与数据结构刷题
- 科学计算生态，如 `numpy / pandas` 的系统化学习
- Web 框架全栈开发
- 并发与异步的深水区

这些不是永远不用学，而是不是当前阶段的瓶颈。先把“能做出可调试的 Agent 原型”这条主线跑通。

## 阶段 0：运行环境与最小脚本能力

### 目标

先具备“能跑起来、能改起来、知道代码在哪执行”的最低能力。

### 学什么

- Python 安装与版本概念
- 虚拟环境
- `pip` 与 `requirements.txt`
- 从命令行运行 `.py` 文件
- 理解 `print`、输入参数、返回值、退出报错

### 为什么这一步重要

如果连环境、依赖和运行方式都不稳定，后面学 API 调用、Agent loop、测试都会被基础问题反复打断。

### 典型产出

- 一个 `hello_python.py`
- 一个读取环境变量并打印配置状态的小脚本
- 一份“如何创建虚拟环境并运行脚本”的笔记

### 完成标志

- 你知道代码是在什么环境里运行的
- 你能独立安装依赖并运行脚本
- 你看到报错时，至少知道先看 traceback

## 阶段 1：最小语法与控制流

### 目标

掌握写脚本和读简单项目时最常见的 Python 语法，不在基础表达上卡住。

### 学什么

- 变量、基本类型、字符串
- 条件判断
- `for` 循环
- 函数定义、参数、返回值
- 常见内置函数，如 `len`、`range`、`enumerate`
- 导入模块与使用标准库

### 学到什么程度就够

标准不是“会背语法”，而是：

- 能看懂一段 50 到 100 行的简单脚本
- 能把重复代码提成函数
- 能根据报错定位到对应函数和参数

### 和 Agent 开发的关系

后续所有 Agent 代码，本质上都离不开条件分支、函数调用、循环控制和模块导入。你不是在学“基础语法”，而是在学之后写 workflow 的最小表达能力。

### 典型产出

- 一个接收用户输入并返回简单结果的脚本
- 一个把多步逻辑拆成 2 到 3 个函数的小练习
- 一份“函数什么时候该拆出来”的笔记

### 完成标志

- 你能读懂简单函数之间的调用关系
- 你能自己写出有输入、有处理、有返回值的脚本
- 你不会再把所有逻辑都堆在一个文件最底部

## 阶段 2：核心数据结构与数据变换

### 目标

掌握处理模型输入输出、工具参数和中间状态最常见的数据结构。

### 学什么

- `list`：顺序数据、遍历、过滤、拼接
- `dict`：键值结构、嵌套访问、更新、默认值处理
- `set`：去重
- `tuple`：简单不可变组合
- 列表推导式和字典推导式，只学常用写法

### 为什么这一步对 Agent 特别重要

Agent 系统里大量数据天然就是这些结构：

- 消息列表
- tool 参数
- structured output
- 检索结果集合
- eval 样本和运行记录

如果这一块不稳，后面写 tool schema、处理模型响应、做日志记录都会很吃力。

### 典型产出

- 一个把消息列表整理成统一格式的小脚本
- 一个把 JSON 样例解析成你需要字段的练习
- 一份“什么时候用 list，什么时候用 dict”的笔记

### 完成标志

- 你能从嵌套 `dict` 里取出需要的信息
- 你能把一组对象整理成新结构
- 你知道很多 LLM 输出处理问题其实是数据结构问题

## 阶段 3：函数拆分、模块组织与可读性

### 目标

从“能写脚本”进到“能把代码写成后面还敢改的样子”。

### 学什么

- 模块与文件拆分
- `if __name__ == "__main__":`
- 函数职责边界
- 变量命名、参数命名、返回值清晰化
- 基础类型标注，只学 `str`、`int`、`list[...]`、`dict[...]`、`Optional[...]` 这些常见形式

### 重点不是规范本身，而是可维护性

后面做 Agent 时，一段代码里往往同时有：

- prompt 组织
- API 调用
- tool 执行
- 结果解析
- 错误处理

如果没有最基本的模块拆分和函数边界，系统很快就会变得难以调试。

### 典型产出

- 把一个单文件脚本拆成 `main.py`、`client.py`、`utils.py`
- 一个带类型标注的最小 API 调用模块
- 一份“什么逻辑该进函数，什么逻辑该留在主流程”的笔记

### 完成标志

- 你能读懂一个最小多文件 Python 项目
- 你能把 100 行左右脚本做一次合理拆分
- 你知道类型标注主要是为了减少理解成本，不是为了炫技

## 阶段 4：I/O、JSON、异常处理与配置

### 目标

补上 Agent 工程里最常见的“和外界打交道”的基础能力。

### 学什么

- 文件读写
- `json` 模块
- 环境变量和 `.env` 的作用
- `try / except / raise`
- 常见异常类型
- 日志与调试打印的最小习惯

### 为什么这是关键分水岭

从这一阶段开始，你写的就不只是“语法练习”，而是接近真实 LLM 应用了。因为后续你会频繁处理：

- API key
- 请求和响应 JSON
- 本地数据文件
- 失败重试
- 输入校验

### 典型产出

- 一个读取本地 JSON 文件并格式化输出的脚本
- 一个带异常处理的配置加载模块
- 一份“哪些错误该直接报，哪些错误该捕获”的笔记

### 完成标志

- 你能看懂大多数基础 traceback
- 你不会把密钥硬编码进脚本
- 你知道错误处理不是为了吞错，而是为了更快定位问题

## 阶段 5：HTTP 请求与模型 API 调用

### 目标

把 Python 基础正式接到 LLM / Agent 主线，完成最小模型调用闭环。

### 学什么

- HTTP 请求的基本概念
- 请求方法、请求头、请求体
- 状态码、超时、重试的基本含义
- 使用 Python SDK 或 HTTP 客户端调用模型
- 解析响应对象

### 这一阶段的重点

不是去背 HTTP 细节，而是搞清楚：

- 你的代码发出了什么请求
- 模型返回了什么结构
- 错误发生在认证、参数、网络还是响应解析

### 典型产出

- 一个最小 LLM 调用脚本
- 一个结构化输出实验
- 一份“调用失败时的排查顺序”笔记

### 完成标志

- 你能独立跑通一次模型调用
- 你能把响应中的关键信息提取出来
- 你知道模型调用不只是 `print(response)`，还要考虑异常和结构

## 阶段 6：面向 Agent 的 Python 实战模式

### 目标

把前面的 Python 基础收束到最接近 Agent 工程的表达模式上。

### 学什么

- 用函数表示工具
- 用 `dict` / 数据对象表示工具参数和运行状态
- 用循环表达最小 agent loop
- 输入校验与结果校验
- 把“模型逻辑”和“工具逻辑”拆开
- 为后续 eval 和调试保留最小日志

### 为什么这一步必须有

很多人学完 Python 语法，还是接不上 Agent。问题不是“不懂类和继承”，而是没有把基础语法映射到 Agent 的常见代码形态上。

你需要开始建立这样的对应关系：

- 函数 = 工具
- `dict` = 消息、状态、schema 对象
- 循环 = agent 执行轮次
- 异常处理 = 工具失败与恢复路径
- 模块拆分 = prompt、tools、runner、utils 的边界

### 典型产出

- 一个本地工具调用 demo
- 一个最小 `agent loop` 原型
- 一份“普通 Python 脚本如何过渡到 Agent 代码”的笔记

### 完成标志

- 你能读懂一个最小 tool-calling demo
- 你能自己实现一个有 1 到 2 个工具的简单 agent 原型
- 你知道代码复杂度主要会先坏在控制流和异常，而不是坏在高级语法

## 推荐学习顺序

建议按下面顺序推进，不要倒着学：

1. 环境和运行
2. 基础语法与函数
3. 数据结构
4. 模块组织
5. 文件、JSON、异常、配置
6. HTTP 和模型调用
7. 面向 Agent 的最小实战模式

如果你跳过前 4 步直接上框架，后面大概率会陷入“看得懂 demo，改不动项目”。

## 每一阶段都应该留下的仓库产出

- `lectures/`：当前阶段最小讲义
- `labs/`：一个单点实验
- `notes/`：踩坑、术语、调试顺序
- `projects/`：每完成 2 到 3 个阶段后做一个组合小项目
- `reviews/`：复盘这轮最常见的错和真正卡点

## 适合这个主题的阶段项目

### 项目 1：最小模型调用器

目标：

- 从命令行读取问题
- 调用模型
- 打印回答
- 处理最基本错误

它对应的能力：

- 环境变量
- 函数封装
- API 调用
- JSON / 响应解析

### 项目 2：工具调用小助手

目标：

- 定义 1 到 2 个本地工具函数
- 让主流程根据输入决定是否调用工具
- 把结果统一格式输出

它对应的能力：

- 控制流
- 数据结构
- 函数边界
- 异常处理

### 项目 3：最小 Agent Runner

目标：

- 支持消息列表
- 支持一次或多次工具调用
- 记录每轮输入、输出和错误

它对应的能力：

- 循环
- 状态管理
- 模块化
- 调试与日志

## 这一主题里的 Codex 协作主线

- 先让 Codex 解释已有 Python 代码在做什么，再改代码
- 让 Codex 帮你把“大脚本”拆成更小函数
- 报错时让 Codex 先给排查顺序，不直接要“重写一版”
- 让 Codex 帮你 review：这段代码以后做 Agent 时哪里最容易坏

不要过早依赖 Codex 做的事：

- 不看代码就整段接受生成结果
- 对异常处理和控制流不做自己的判断
- 把项目复杂度问题误以为是“再换个 prompt 就好”

## 学完这个主题后，你应该具备什么能力

- 能读懂后续大部分入门级 Agent 教程代码
- 能自己改动最小 LLM / tool-calling 项目
- 能看懂常见报错并做第一轮排查
- 能开始进入后续的 `LLM Fundamentals`、`RAG` 和 `AI Agents` 主题，而不会一直被 Python 基础卡住

## 判断是否可以进入下一主题

只有满足下面条件，才建议进入后续 `LLM Fundamentals`：

- 你能独立运行一个 Python 项目
- 你能调用一次模型 API 并拿到结果
- 你能处理基础 JSON、文件和异常
- 你能把简单逻辑拆成函数和模块
- 你能读懂一个最小 tool-calling demo 的 Python 结构

如果还做不到这些，不要继续扩大学习范围，先回到当前主题补最小闭环。
