## Lecture 理解

- 为什么 Agent 项目里更容易出现“一个文件越来越胖”：
  因为 Agent 项目天然会不断往里加新环节：模型调用、prompt 组装、tool 调用、数据清洗、日志、配置、异常处理。入门时这些逻辑都能先堆在一个脚本里跑通，但后面每加一层能力，如果不主动拆模块，旧文件就会继续吸收新职责，最后变成“什么都沾一点”的大杂烩。它之所以容易胖，不是因为 Python 特别差，而是因为 Agent workflow 本来就是多步骤组合流程。
- `main.py` 该保留什么，不该保留什么：
  `main.py` 应该保留入口职责：读取输入、调用几个关键步骤、串起主流程、输出结果，必要时处理最外层异常。它不该保留业务细节，比如具体的数据转换规则、prompt 组装细节、工具实现、复杂分支判断；这些都应该下沉到更具体的模块里。一个实用判断是：`main.py` 应该让人一眼看懂“流程怎么走”，而不是逼人读懂“每一步怎么实现”。
- 类型标注什么时候有帮助，什么时候只是噪音：
  当类型标注能明显降低理解成本时，它有帮助，比如函数参数和返回值不再需要靠猜、一个值是单个对象还是列表一眼能看出来、模块之间的调用契约更清楚。反过来，如果标注只是把本来就显而易见的东西机械重复一遍，或者为了追求“类型完整”写出很长很绕的类型，导致读代码的人反而更累，那它就是噪音。现阶段最有价值的是最小类型标注：先标函数边界，而不是沉迷把所有局部变量都写满。
- 我当前认可的模块拆分原则：
  先按职责拆，让一个模块只负责一类变化，例如输入输出、数据转换、模型调用、工具逻辑分开。
- 哪些类型标注对我有帮助：
  参数类型、返回值类型、以及容易混淆的数据结构类型最有帮助。
- 哪些写法只是形式主义：
  对显而易见的局部变量过度标注，或者写出很复杂但几乎没人能快速读懂的类型声明，通常只是形式主义。
