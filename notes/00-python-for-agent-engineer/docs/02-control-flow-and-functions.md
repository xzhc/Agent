## Lecture 理解

- 为什么“能跑”不等于“后面还敢改”：
  代码这次能跑，只说明当前输入刚好走通；但如果逻辑混在一起、输入输出边界不清、状态靠隐式共享传递，后面一改分支就容易牵一发而动全身。真正“还敢改”的前提，是你能定位每段逻辑负责什么、改动后影响哪些地方、以及能不能单独验证。
- 什么时候应该拆出新函数：
  当一段代码开始承担独立职责时就该拆，比如“分类”“格式化”“输出”已经是三个不同动作；或者你发现它开始被重复使用、分支越来越多、名字越来越难起、没法一句话说清它在做什么，也说明该拆了。一个实用判断是：如果这段逻辑值得被单独命名，它通常就值得单独成函数。
- 如果一个函数既做分类、又做格式化、又做输出，它后面会怎么坏：
  最常见的坏法是修改一个环节时误伤另一个环节。比如你只是想改分类规则，却把输出格式一起改坏；或者你想复用分类结果，却发现函数内部已经直接 `print` 了，拿不到可组合的返回值。结果通常是测试难写、复用困难、调试时看不清是哪一层出错，最后函数越补越长、越来越不敢动。
- 什么时候该拆函数：
  看到职责混合、重复逻辑、超长分支、输入输出说不清时就该拆。
- 我最容易把逻辑堆在哪里：
  最容易堆在 `main()`、一个大 `if/elif`、或者“先写着再说”的工具函数里。
- 这讲最关键的一句判断：
  先让函数只做一件事，并且优先返回结果，而不是顺手把别的事也做了。
