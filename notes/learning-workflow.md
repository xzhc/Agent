# 学习流程

这份文档定义这个仓库的默认学习方式。目标不是“看懂资料”，而是稳定形成两类能力：

- Agent 工程能力：会设计、实现、调试、评估和迭代真实可用的 Agent 系统。
- AI 提效能力：会在真实工作中使用 Codex、MCP 和 skills 提高效率，并知道什么时候该用、什么时候不该用。

这两条主线是并行推进的，不是先学完 Agent 再回头学工具，也不是先装一堆工具再找地方使用。

## 核心原则

- 先理解最小必要知识，再动手验证。
- 先做单点实验，再做组合项目。
- 每学完一轮都要留下可复用产出，而不是只留下聊天记录。
- 不只关注“怎么做”，还要关注“为什么这样做”和“哪里容易坏掉”。
- 关于 Codex、MCP 和 skills 采用“干中学”，围绕真实任务渐进式引入。

## 双主线学习闭环

这个仓库默认不是单一学习链路，而是双主线闭环：

1. `lectures/`：先学当前主题的最小必要知识。
2. `labs/`：做单点实验，把这个能力真正跑通。
3. `notes/`：记录这次学习中稳定有复用价值的理解。
4. `projects/`：把多个能力组合成更接近真实工作的项目。
5. `reviews/`：阶段结束后做复盘，明确能力边界和薄弱点。
6. `resources/`：在需要时补充官方资料、开源项目和工程实践文章。

上面这条是仓库的结构化学习流程。它内部又同时承载两条主线：

- Agent 工程主线：学习模型调用、tool use、memory、planning、evals、工程化等能力。
- AI 提效主线：学习如何与 Codex 协作，以及什么时候按需引入 MCP、安装 skill 或创建 skill。

这不是线性一次性流程，而是一个循环。每完成一个主题，都会回到下一轮新的 `lecture -> lab -> notes -> project/review`。

## 每个目录在双主线中的职责

### `lectures/`

作用：

- 帮你建立当前主题的最小认知框架。
- 解释“这是什么、为什么需要、在真实项目里解决什么问题”。
- 在合适的时候补充“这个场景里 Codex 能怎么帮你”。

完成标准：

- 能用自己的话解释核心概念。
- 知道这个主题在 Agent 系统里的位置。
- 知道最常见的失败方式是什么。
- 知道在当前主题里，哪些步骤适合与 Codex 协作。

### `labs/`

作用：

- 用最小代码验证一个知识点是否真的理解了。
- 把“看懂”变成“跑通”。
- 在真实编码、调试和验证中练习如何与 Codex 配合。

完成标准：

- 代码能运行。
- 知道预期输入和输出。
- 遇到常见错误时能自己排查一轮。
- 知道 Codex 在这个实验里帮你加速了哪一部分，哪些判断仍然需要你自己做。

### `notes/`

作用：

- 沉淀长期有价值的理解，而不是重复聊天内容。
- 形成自己的术语表、踩坑清单和调试套路。
- 记录哪些 Codex 协作方式真正有效，哪些只是看起来省事。

完成标准：

- 记录了容易忘、容易错、容易在真实项目中反复出现的问题。
- 记录了某些场景下与 Codex 协作的有效方式。
- 后续做项目时能反过来查这些笔记。

### `projects/`

作用：

- 把多个单点能力组合成真正可工作的 Agent 系统。
- 开始面对真实工程问题，比如状态、异常、评估、成本和可靠性。
- 在项目里判断什么时候该引入更稳定的 AI 工作流。

完成标准：

- 项目能完成一个明确用户任务。
- 不只是 demo，而是有基本结构、运行方法和已知问题说明。
- 能说清楚这个项目里 Codex 的协作边界。

### `reviews/`

作用：

- 复盘这轮学习和项目的收获与问题。
- 找出“不会”的真实原因，而不是模糊地觉得自己没学会。
- 判断哪些高频动作已经值得沉淀成更稳定的工作流、skill 或外部连接。

完成标准：

- 说清楚卡点、根因和下一步补强方向。
- 能形成下一轮学习计划。
- 能说明哪些与 Codex 的协作方式值得保留。

### `resources/`

作用：

- 提供高质量外部资料的索引。
- 帮你在合适的时候对齐一线实践，而不是一开始被海量资料淹没。

完成标准：

- 知道该看哪份资料、为什么看、重点看哪里。
- 知道某份资料是补 Agent 工程能力，还是补 Codex / MCP / skills 的工作方式。

## 一轮典型学习流程

以 `tool calling` 为例，一轮完整学习不只是学概念，还要顺手建立对应的 AI 协作方式：

1. 在 `lectures/` 学清楚 tool calling 的概念、作用、常见错误和最小实现方式。
2. 在 `labs/` 做一个只验证 tool calling 的小实验，比如让模型调用一个本地函数。
3. 在 `notes/` 记录这次实验中暴露出来的关键点，比如 schema 设计、参数校验、失败重试。
4. 在 `projects/` 把它放进一个更真实的 Agent 里，比如“查询信息并整理输出”的工作流。
5. 在 `reviews/` 复盘：这次做不好到底是 prompt 设计问题、工具设计问题，还是异常处理问题。
6. 如果需要更高质量参考，再去 `resources/` 看 OpenAI、Anthropic 或开源项目里的实践。

与这条 Agent 工程链路并行的 Codex 提效链路通常是：

1. 先让 Codex 帮你拆解这个实验要验证什么，不直接要一大段最终代码。
2. 编码时让 Codex 先读已有文件，再做最小修改，而不是无上下文地重写。
3. 报错时让 Codex 协助缩小问题范围，给出排查顺序。
4. 复盘时记录：这次哪些事情交给 Codex 很有效，哪些事情必须自己判断。

## MCP 和 skills 的引入方式

### 什么时候引入 MCP

只有在出现下面场景时，才建议开始学习或接入 MCP：

- 当前任务需要稳定访问外部系统，而手工复制粘贴上下文已经明显低效。
- 需要从某个系统持续获取结构化信息或执行操作。
- 没有 MCP 时，Codex 反复依赖你手工搬运信息，已经拖慢了工作流。
- 这个连接会在多个任务里复用，而不是一次性操作。

暂时不建议引入 MCP 的场景：

- 当前问题只靠本地代码和现有工具就能解决。
- 你还在入门阶段，连最小流程都没有跑通。
- 只是觉得“以后可能会用到”，但现在没有稳定需求。

### 什么时候安装已有 skill

只有在出现下面场景时，才建议安装已有 skill：

- 某类任务会反复出现，而且步骤相对稳定。
- 已经有成熟的最佳实践、模板或检查清单，复用收益明显。
- 这个能力跨项目、跨仓库都可能复用。

### 什么时候自己创建 skill

只有在出现下面场景时，才建议创建新 skill：

- 某类任务在本仓库或你的工作流里反复出现，但现有 skill 不贴合。
- 你已经摸索出一套相对稳定的步骤、判断标准或模板。
- 这套流程值得沉淀，而不是每次重新口头描述。

暂时不建议创建 skill 的场景：

- 任务只做一次。
- 工作流还没稳定，经常变。
- 你自己还没有验证过什么做法真正有效。

## 推荐节奏

如果当前是从零开始，建议按下面节奏推进：

1. 每次只学一个主题，不并行开太多坑。
2. 每个主题至少配一个 `lab`，避免只看讲义。
3. 每学完 2 到 3 个主题，就做一个小项目。
4. 每完成一个项目，就做一次复盘。
5. 每次卡住时，先补最小必要资料，不要无限扩展阅读。
6. 不预装大量 MCP 或 skills，等到真实痛点出现时再引入。

## 什么时候可以进入下一阶段

只有满足下面条件，才算一个主题基本学完：

- 你能讲清楚这个主题的作用。
- 你能独立跑通最小实验。
- 你知道它最常见的坑。
- 你能把它放进一个更大一点的项目里。
- 你知道在这个主题里，Codex 最适合帮你做什么。

如果只是“看懂了讲义”，还不算真正学会。如果只是“会让 Codex 生成代码”，也不算真正学会。

## 当前仓库的建议起点

当前仓库已经有基础目录结构，下一步建议是：

1. 在 `lectures/` 创建第一份讲义：`00-minimum-agent/`
2. 在 `labs/` 创建第一个实验：`lab-01-first-llm-call/`
3. 用第一轮最小闭环跑通“从调用模型到完成一个简单任务”
4. 在第一轮内容里同步加入“如何与 Codex 协作完成最小任务”

## 使用建议

- 每次开始新主题前，先看这份文档，确认当前应该处在哪个环节。
- 如果发现自己一直在读资料但没有产出，直接回到 `labs/`。
- 如果发现自己一直在写代码但没有总结，回到 `notes/` 和 `reviews/`。
- 如果发现自己做的都是碎片练习，说明该进入 `projects/` 了。
- 如果发现你频繁重复同一类手工操作，再回头判断是否该引入 MCP 或 skill。
